# 국내 의료데이터 Azure 클라우드 이관 — 컴플라이언스 가이드

> ⚠️ 본 문서는 **설계 검토용 요약**이며 법률 자문이 아니다.
> 실제 배포 전 최신 조문(국가법령정보센터)·복지부 가이드라인 확인과 **법무·IRB/DRB 검토**가 필수다.

## 1. 핵심 요지

1. **용도가 곧 법적 근거** — 진료(1차)와 연구(2차)는 근거법·동의·데이터형태·심의 절차가 완전히 다르다.
2. **데이터는 물리적으로 분리** — 진료(실명)와 연구(가명) 영역을 구독/네트워크 단위로 분리, 단방향 게이트.
3. **국내 리전 · CSAP · 암호화** — Korea 리전 고정, 공공/의료는 CSAP 범위 내, CMK 암호화·최소권한·2년 로그.
4. **재식별 차단이 핵심 통제** — 가명처리 후 반입, 매핑키 분리 보관, 재식별·역방향 경로 원천 차단.

## 2. 적용 핵심 법령·기준

| 법령 / 기준 | 규율 대상 핵심 |
| --- | --- |
| 개인정보보호법 (PIPA) | 건강정보=민감정보. 별도 안전조치·동의. §28-2 가명정보 특례 |
| 의료법 §23 / 시행규칙 §16 | 전자의무기록(EMR) 작성·보관, 외부(클라우드) 보관 시설·장비 요건 |
| 생명윤리 및 안전에 관한 법률 | 인간대상·인체유래물 연구 시 IRB 심의, 유전정보 활용 제한 |
| 보건의료데이터 활용 가이드라인 | 가명처리 방법·데이터심의위원회(DRB) 절차 (보건복지부) |
| 클라우드컴퓨팅법 / CSAP | 공공·의료 클라우드 보안인증(등급·리전·배포유형 단위) |

## 3. 용도별 비교 — 진료 vs 연구

| | 진료 목적 (1차) | 연구 목적 (2차) |
| --- | --- | --- |
| 근거 | 의료법·PIPA §15 (진료계약·법령 의무) | PIPA §28-2·생명윤리법 |
| 동의 | 진료 목적 처리는 별도 동의 불요 | 동의 기반(+IRB) 또는 가명처리 기반(동의 면제) |
| 데이터 | 실명(식별) 데이터 | 가명/익명 (동의 시 실명) |
| 심의 | 없음 | IRB / DRB |
| 의무 | EMR 외부보관 요건·암호화·접근통제·접속기록 2년+ | 가명처리·재식별 금지·결합전문기관 경유 |

- **트랙 A (동의 기반)**: 명시적 동의 + IRB 심의 → 식별데이터 연구 활용 가능.
- **트랙 B (가명처리 기반, PIPA §28-2)**: 과학적 연구·통계·공익 목적 한정, 동의 면제.
  ① 가명처리 → ② DRB 승인 → ③ 재식별 금지 → ④ 결합은 결합전문기관 경유.
- ⚠️ 진료용 데이터를 연구에 재사용 = **목적 외 이용** → 동의 또는 가명처리+DRB 필요 (자동 허용 아님).

## 4. 목적별 분리 아키텍처 (단방향 게이트)

```
┌─ 진료 영역 (식별 데이터) ────┐        ┌─ 연구 영역 (가명 데이터) ────┐
│ Subscription/RG: clinical    │        │ Subscription/RG: research    │
│ - Azure Health Data Services │  가명  │ - 가명처리 후 데이터만 반입   │
│   (FHIR)                     │  처리  │ - Synapse/Databricks 분석    │
│ - 실명 EMR                   │ ─────▶ │ - AI/ML 학습                 │
│ - 접근: 진료 인가자          │  게이트│ - 접근: 연구 승인자(DRB)     │
│ - 매핑키 Key Vault 격리      │ (단방향)│ - 재식별키 도달 불가         │
└──────────────────────────────┘        └──────────────────────────────┘
```

**핵심**: 진료 → (가명처리) → 연구 **단방향만** 허용. 역방향·재식별 경로 원천 차단.
매핑키는 진료영역에 분리 보관, 연구자 접근 불가.

### 왜 역방향(연구→진료)은 원천 차단인가

"역방향" = ① 연구 데이터를 진료로 되돌리기 ② 가명데이터를 실명으로 복원(재식별).

1. **재식별 금지(위법)** — PIPA §28-5. 경로가 열리면 '매핑키+가명데이터' 결합으로 복원 가능.
2. **목적 구속 위반** — 연구 데이터는 '동의 없이 연구 목적에 한해' 허용된 것. 진료로 되돌리면 처리 근거 붕괴.
3. **기록 무결성 훼손** — 연구영역은 가공·AI 학습 영역. 진료기록 역류 시 EMR 원본성 훼손(의료법).
4. **재식별은 실제 위협** — 가명데이터도 타 정보 결합 시 식별 가능. 단방향+매핑키 격리가 결합 자체를 차단.

> 연구 성과를 진료에 쓰려면? 데이터 역류가 아니라 **모델·통계(비개인정보)만 전달** →
> 진료영역에서 새 동의 기반으로 적용.

## 5. 데이터가 올라가는 Azure 서비스 (용도별)

### 진료 영역 (실명·식별)

| 서비스 | 용도 |
| --- | --- |
| Azure Health Data Services (FHIR/DICOM/MedTech) | EMR·영상·기기 데이터 표준 저장·교환 |
| Azure SQL DB / SQL MI · PostgreSQL | 진료·처방 등 정형 데이터 |
| Blob Storage (ADLS Gen2) | 영상 원본·문서·첨부 |
| API Management · Service Bus / Event Hubs | 원내 시스템·기기 연계 |
| Azure Cache for Redis | 조회 성능·세션 |

### 연구 영역 (가명·익명)

| 서비스 | 용도 |
| --- | --- |
| ADLS Gen2 (Data Lake) | 가명처리 데이터 적재 |
| Databricks · Synapse · Microsoft Fabric | 대규모 분석·가명처리 파이프라인 |
| Azure Data Factory | 진료→가명처리→연구 이동(ETL) |
| Azure Machine Learning | 모델 학습·실험 관리 |
| Azure OpenAI · AI Foundry · AI Search | RAG·LLM (가명데이터 기반) |

### 공통 보안·거버넌스

| 범주 | 서비스 |
| --- | --- |
| 키/암호화 | Key Vault / Managed HSM (CMK, 재식별키 격리) |
| ID/권한 | Entra ID · RBAC · PIM · Conditional Access |
| 네트워크 | Private Endpoint / Private Link · Firewall · NSG |
| 감사/모니터 | Log Analytics · Monitor · Sentinel |
| 데이터 거버넌스 | Microsoft Purview (분류·계보·민감도 라벨) |
| 정책 | Azure Policy (리전 잠금·태그 강제) |

### 서비스 선택 시 필수 확인

1. **CSAP 인증 범위 포함 여부** (IaaS 중심, PaaS/AI는 서비스별 상이).
2. **국내 리전(Korea Central/South) 및 기능 제공 여부**.
3. **데이터 잔류 위치** — AI·검색 인덱스·로그의 리전 밖 유출 여부.
4. **Private Link 지원** (폐쇄망 구성 가능 여부).
5. **CMK(고객관리키) 지원** (저장 암호화 키 통제).

## 6. PIPA §28-5 — 가명정보 재식별 금지 (핵심 조항)

- ① 누구든지 **특정 개인을 알아보기 위한 목적**으로 가명정보를 처리해서는 안 된다.
- ② 처리 과정에서 특정 개인을 알아볼 수 있는 정보가 생성되면 **즉시 처리를 중지하고 지체 없이 회수·파기**해야 한다.
- '누구든지' = 처리자·연구자·수탁자·제3자 모두 대상.
- **위반 시**: 재식별 목적 처리 시 5년 이하 징역 또는 5천만원 이하 벌금(§71), 과징금·시정명령.

관련 조문(가명정보 특례): §28-2(동의 없이 처리) · §28-3(결합전문기관) · §28-4(안전조치) · §28-5(재식별 금지).

## 7. 사전 체크리스트

**진료 영역**
- [ ] EMR 외부보관 요건(시행규칙 §16)·신고 대상 확인
- [ ] 실명데이터 접근 = 진료 인가자로 한정 (RBAC·조건부 액세스)
- [ ] Health Data Services(FHIR) 인증범위·리전 확인
- [ ] 백업·DR 국내 리전 내 유지
- [ ] 개발/테스트에 운영 실명데이터 복사 금지 (가명·합성데이터 사용)

**연구 영역**
- [ ] 진입 전 가명처리 완료 (식별데이터 직접 반입 금지)
- [ ] DRB/IRB 승인 문서 확보
- [ ] 매핑키 진료영역 격리, 연구자 접근 차단
- [ ] 결합 필요 시 결합전문기관 경유
- [ ] AI 학습 결과·모델 재식별 잔존 여부 검토
- [ ] 유전·생체정보 포함 시 별도 법적 검토

**자주 놓치는 함정**
- 모니터링/텔레메트리가 글로벌 리전으로 유출 → 진단 데이터 옵트아웃 확인
- 재식별키를 같은 볼트·같은 구독에 보관 → 물리·논리 분리 필수
- 백업이 CSAP 인증 범위·국내 리전 밖에 존재
- 진료 데이터를 동의·DRB 없이 연구/AI 학습에 재사용 (목적 외 이용)

## 8. 국내법 충족 여부 (요약)

기술 설계는 국내법이 요구하는 **기술적 안전조치를 충족할 수 있는 구조**이나,
**"법 만족 = 기술 + 절차 + 계약"**. 다음이 병행되어야 완전 충족:

1. IRB/DRB 승인 (연구)
2. Microsoft와 DPA(위수탁) 체결
3. EMR 외부보관 신고 대상 확인
4. CSAP 인증서 서비스 범위 대조
5. 가명처리 적정성 검증
